Acceptance Criteria & Smoke Test Guide

Comprehensive QA checklists for all components and user experiences in Smart App 2.0

This document provides comprehensive acceptance criteria (AC) for all common components and user experiences in Smart App 2.0. Use this as a checklist for QA, smoke testing, and ensuring production-ready quality.

üéØ User Experience Acceptance Criteria

UX Principle

Navigation & Wayfinding

Users should always know where they are and how to get back

  • Breadcrumb trail shows complete path from Home to current page
  • All breadcrumb links are functional
  • Back button returns to correct parent page
  • Current page indicated in breadcrumbs (non-clickable, teal color)
  • Navbar shows current section as "active"
  • Footer includes contextual next/previous links
  • URL structure reflects content hierarchy
UX Principle

Consistency & Predictability

Every page should follow established patterns

  • Navigation bar appears at top of every page
  • Breadcrumbs appear below navbar on detail pages
  • Page header format consistent across sections
  • Back button in same position on all detail pages
  • Footer appears at bottom of every page
  • Color scheme matches design system
  • Typography hierarchy consistent (h1, h2, h3 usage)
UX Principle

Responsive Design

Works perfectly on mobile, tablet, and desktop

  • Mobile (320px-767px): Single column, stacked elements
  • Tablet (768px-1023px): Adjusted grid, readable text
  • Desktop (1024px+): Full layout, all features visible
  • Touch targets minimum 44x44px on mobile
  • Text readable without zoom on all devices
  • Images scale appropriately
  • Tables scroll horizontally on small screens
  • No horizontal scroll on any breakpoint
UX Principle

Performance & Loading

Fast, smooth experience with clear feedback

  • Page loads in <2 seconds on 3G connection
  • Images lazy-load below fold
  • No render-blocking resources
  • Smooth scroll behavior enabled
  • Animations perform at 60fps
  • Loading states for async operations
  • No flash of unstyled content (FOUC)
UX Principle

Accessibility

Usable by everyone, including assistive technology

  • Semantic HTML elements used correctly
  • ARIA labels on interactive elements
  • Keyboard navigation works throughout
  • Focus indicators visible and clear
  • Color contrast meets WCAG AA standards (4.5:1)
  • Alt text on all images
  • Form labels properly associated with inputs
  • Screen reader announces page changes

üß© Component-Level Acceptance Criteria

Component

Navigation Bar (Navbar)

Location: components.css lines 1-71

Functional Acceptance Criteria:

  • Navbar sticky at top when scrolling
  • Logo/brand links to Home (index.html)
  • All nav links navigate to correct sections
  • Active section highlighted visually
  • Hover states work on all links
  • Mobile: Hamburger menu icon appears <768px
  • Navbar doesn't overlap content

Visual Acceptance Criteria:

  • Background color: --bg-card (#1a2234)
  • Text color: --text-primary (#f1f5f9)
  • Active link color: --accent-teal (#14b8a6)
  • Navbar brand icon displays correctly
Component

Breadcrumbs

Location: components.css lines 73-143

Functional Acceptance Criteria:

  • All non-current links are clickable
  • Current page shown but not clickable
  • Separator ‚Ä∫ between each level
  • Breadcrumb path matches URL hierarchy
  • Works at all hierarchy levels (2-level, 3-level)

Visual Acceptance Criteria:

  • Links: --accent-blue with hover effect
  • Current page: --text-muted
  • Separators: --text-muted
  • Container max-width: 1400px
Component

Data Tables

Used throughout for requirements, features, research data

Functional Acceptance Criteria:

  • Table scrolls horizontally on mobile
  • Headers are sticky on scroll (optional)
  • Status badges display correctly
  • Links in tables work properly
  • Zebra striping helps readability

Visual Acceptance Criteria:

  • Full width with rounded corners
  • Header background: --bg-secondary
  • Row hover effect visible
  • Cell padding consistent (1rem)
  • Border color: --border-color
Component

Status Badges

Color-coded status indicators

Badge Variants:

  • .status-complete - Green background (#10b981)
  • .status-in-progress - Yellow background (#f59e0b)
  • .status-pending - Gray background (#64748b)
  • .badge-success - Green outline
  • .badge-info - Blue outline
  • .badge-warning - Yellow outline

Visual Acceptance Criteria:

  • Rounded corners (4px border-radius)
  • Padding: 0.25rem 0.75rem
  • Font size: 0.85rem
  • Font weight: 500
  • Text transforms to uppercase

‚úÖ Quality Standards

Quality Standard

Code Quality

Standards for HTML, CSS, and JavaScript

  • HTML validates (no errors in W3C validator)
  • CSS classes follow naming convention
  • No inline styles except for dynamic values
  • JavaScript has error handling
  • Console has no errors or warnings
  • No deprecated HTML tags or attributes
  • Code is readable with proper indentation
  • Comments explain complex logic
Technical Standard

Deployment & Version Control

Git and Vercel standards

  • Commit messages are descriptive
  • Code reviewed before merging
  • Vercel deployment succeeds
  • No broken links in production
  • All images load correctly
  • HTTPS works correctly
  • Vercel Analytics enabled

üî• Quick Smoke Test Checklist

Run this quick checklist after every deployment to catch critical issues:

5-Minute Test
  • Homepage loads without errors
  • All main navigation links work
  • Breadcrumbs function correctly
  • Back buttons return to parent page
  • Charts render on homepage
  • Mobile view doesn't break layout
  • Footer links work
  • No console errors
  • Images load properly
  • Search functionality works (if enabled)
