Executive Summary

Co-Borrower Invitation Methods

Smart App 2.0 supports two distinct invitation pathways, optimized for different origination scenarios and borrower preferences. Both methods provide identical collaboration capabilities once co-borrower joins.

Method 1: Borrower-Initiated Invitation

Primary borrower starts application solo
System detects co-borrower needed (loan purpose, income requirements)
Borrower sends invitation via email/SMS from within app
Co-borrower creates own account with independent OTP auth
Both borrowers gain immediate access to shared application
Self-Service 62% of POC co-borrower apps

Method 2: LO-Initiated Dual Invitation

LO creates application with both borrower contacts
System sends simultaneous invitations to both borrowers
Each borrower receives unique secure link
Both authenticate independently (separate OTP codes)
Application collaboration begins immediately
LO-Driven 38% of POC co-borrower apps

Security Note:

Each co-borrower receives independent OTP codes and maintains separate authenticated sessions. Borrowers cannot access each other's credentials or login as the other person. This satisfies audit requirements for individual verification and provides clear accountability for data entry.

Independent Authentication System

Unlike traditional applications where borrowers share credentials or only one person can log in, Smart App 2.0 provides each borrower with their own secure, phone-verified identity and session management.

Separate OTP Codes

Each borrower receives OTP codes sent to their own phone number. Co-borrower cannot use primary borrower's codes, and vice versa.

Implemented Prove API

Individual Sessions

Each borrower has their own authenticated session with unique session token. Sessions expire independently based on activity.

Implemented JWT-based

Device Fingerprinting

System tracks which device each borrower uses for login. Flags suspicious activity if same device claims to be both borrowers.

Conflict Resolution

When both borrowers attempt to edit the same field or section, Smart App 2.0 employs intelligent conflict prevention and resolution strategies to prevent data loss and maintain data integrity.

Conflict Scenarios & Resolution

Conflict Type Detection Method Resolution Strategy User Experience
Simultaneous Section Editing WebSocket tracks who enters edit mode first First borrower gets lock, second sees lock indicator Second borrower: "Michael is editing this section. It will unlock in 9:32"
Offline Edits (Conflicting Saves) Compare timestamps when both save same field Show both values, ask user to choose correct one Modal: "You entered $5,000 but Sarah entered $5,500. Which is correct?"
Duplicate Asset Addition Match account numbers, property addresses Prevent duplicate, show existing entry "This account was already added by Sarah. View it in Joint Accounts."
Contradicting Answers Logic validation (e.g., both claim to be primary wage earner) Flag for review, notify LO "Both borrowers indicated different answers. Your LO will clarify."
Shared Section Deletion Track who deleted and when Require confirmation from both borrowers "Sarah deleted Property Details. Confirm you agree to remove this section."
Session Timeout Conflict Session expires while editing Save draft locally, restore on re-login "Your session expired. We saved your changes. Click Continue to resume."

Section Locking Mechanism

How It Works:
When Borrower A clicks "Edit" on a shared section, system creates lock record with timestamp
Borrower B attempting to edit sees lock message with countdown timer
Lock auto-releases after 10 minutes of inactivity (no field changes)
Borrower A can manually release lock by clicking "Save" or "Cancel"
LO can override lock if borrowers are stuck (admin function)
WebSocket broadcasts lock status changes in real-time
POC Result:

Section locking prevented 100% of potential data conflicts in POC testing. Zero instances of "last save wins" data loss. Average lock duration was 3.2 minutes. Only 2 out of 487 applications required LO intervention to manually release a stuck lock (both due to browser crashes).

Data Synchronization & Locking Mechanisms

Real-Time Sync Architecture

WebSocket Layer

Persistent WebSocket connection for each active session. Broadcasts state changes to all connected clients in <100ms.

wss://app.smartapp.com/ws/application/{app_id}

Event-Driven Updates

Client subscribes to relevant events (section updates, presence changes). Server pushes updates only when subscribed events occur.

Pub/Sub Pattern

Conflict-Free Replicated Data (CRDT)

Certain fields use CRDT algorithms to allow simultaneous edits without conflicts. Last-write-wins for simple fields.

Future Enhancement

Locking Strategy

Lock Type Scope Duration Auto-Release Override
Section Lock Entire section (e.g., "Property Details") While borrower in edit mode 10 minutes inactivity LO can force release
Field Lock (Future) Individual field within section While field has focus 30 seconds after blur Not applicable (too granular)
Document Lock Uploaded document being viewed/edited While viewer has document open 5 minutes inactivity Not needed (read-only for borrowers)
Application Lock Entire application during final submission Until submission complete or fails 2 minutes (if submission hangs) LO can cancel submission

Sync Edge Cases

Handling Network Partitions:
Detection: Client detects WebSocket disconnect, shows "Offline" indicator
Local Changes: Continue allowing edits, save to localStorage with timestamp
Reconnection: Auto-reconnect with exponential backoff (1s, 2s, 4s, 8s, max 30s)
Sync on Reconnect: Send local changes, receive server state, resolve conflicts if any
Conflict Resolution: If server has newer data, show modal asking user to choose

Key Implementation Patterns

Pattern Use Case Implementation Benefit
Optimistic Updates Form field changes Update UI immediately, sync to server in background Instant feedback, perceived performance
Debounced Sync Typing in text fields Wait 500ms after last keystroke before syncing Reduce server load, batch updates
Event Sourcing Audit trail, history tracking Store every state change as immutable event Complete audit trail, replay capability
Distributed Locks Section editing conflicts Redis-based locks with TTL and auto-release Prevent concurrent edit conflicts
Server-Sent Events (Future) One-way updates from server Fallback for WebSocket if firewall blocks Better compatibility, graceful degradation
CQRS Read-heavy operations Separate read models optimized for queries Faster dashboard loads, better scalability

Database Schema Highlights

Key Tables:
applications - Main application record with status, timestamps, LO assignment
borrowers - Individual borrower records linked to application, stores identity verification
sections - Section completion status per borrower, progress tracking
field_values - Key-value store for form data with versioning, borrower attribution
section_locks - Active locks with borrower_id, timestamp, auto-release timer
events - Event sourcing log, every state change with actor, timestamp, payload
notifications - Notification queue with delivery status, channel, priority
messages - Chat messages between borrowers with read receipts

Performance Optimizations

Lazy Loading Sections

Only load section data when borrower navigates to it. Reduces initial page load by 73%.

WebSocket Connection Pool

Reuse connections across server instances with sticky sessions. Handles 10K+ concurrent users.

Cached Progress Calculations

Cache progress % in Redis, invalidate on section completion. Instant dashboard loads.
