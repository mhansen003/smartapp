Decision Framework

We use the DACI framework (Driver, Approver, Contributors, Informed) for most product and technical decisions, ensuring clear ownership and accountability. For architectural decisions, we follow the ADR (Architecture Decision Record) format to capture context, options, and consequences.

Driver (D)
Responsible for gathering input, driving consensus, and ensuring the decision is made.
Approver (A)
Has final authority to make the decision. Typically a single person to avoid decision paralysis.
Contributors (C)
Provide input and recommendations based on their expertise and perspective.
Informed (I)
Notified of the decision after it's made. Need to know but don't contribute.

ADR-001
Implemented

Multi-Tenant Architecture Approach

Date: 2025-11-15
Approver: CTO, VP Engineering
Impact: High
Context
Smart App 2.0 needs to serve multiple lenders with different branding, configurations, and compliance requirements. We need to decide on a multi-tenancy strategy that balances isolation, performance, cost, and operational complexity.
Options Considered
  • Option A: Separate Database per Tenant - Complete isolation but high operational overhead and cost
  • Option B: Shared Database with Schema per Tenant - Good isolation with moderate complexity
  • Option C: Shared Database with Tenant ID Column - Simple but risk of data leakage
Decision
Option C: Shared Database with Tenant ID Column + Row-Level Security

We chose this approach with PostgreSQL Row-Level Security (RLS) policies to enforce tenant isolation at the database level. This provides simplicity, cost efficiency, and strong security guarantees without the operational overhead of managing hundreds of databases or schemas.
Rationale
‚Ä¢ Operational simplicity - single database to manage, backup, and maintain
‚Ä¢ Cost effective - shared infrastructure with excellent resource utilization
‚Ä¢ PostgreSQL RLS provides database-enforced isolation preventing data leakage
‚Ä¢ Easier to implement cross-tenant analytics and reporting
‚Ä¢ Proven pattern used by companies like Slack and GitHub at scale
Consequences
Positive: Lower operational costs, simpler migrations, easier analytics
Negative: Must be extremely careful with tenant_id filtering in all queries, potential for noisy neighbor issues
Risks: Data leak if RLS policies are misconfigured - mitigated with automated testing and policy review
ADR-002
Implemented

Microservices vs Monolith Architecture

Date: 2025-11-18
Approver: CTO, VP Engineering
Impact: High
Context
Smart App 2.0 integrates with multiple third-party services (Plaid, TRUV, Prove), has complex business logic, and needs to scale independently for different components. Team is currently 8 engineers and will grow to 15-20.
Decision
Modular Monolith with Service Boundaries

Start with a well-structured modular monolith that can be decomposed into microservices later. Use clear module boundaries, shared interfaces, and domain-driven design principles to enable future extraction of services.
Rationale
‚Ä¢ Faster development velocity for current team size
‚Ä¢ Easier debugging and testing with single deployment unit
‚Ä¢ Lower operational complexity (no service mesh, distributed tracing setup initially)
‚Ä¢ Clear module boundaries allow extraction to microservices when needed
‚Ä¢ Proven approach - companies like Shopify and GitHub started this way
Future Migration Path
Extract services starting with:
1. Third-party integration service (Plaid, TRUV, Prove)
2. Document processing service
3. Notification service
4. Analytics/reporting service

Caching Strategy with Redis

Date: 2025-11-22
Approver: Principal Engineer
Impact: Medium
Decision

Use Cases

ADR-005
Implemented

Real-Time Communication: WebSocket vs SSE

Date: 2025-11-25
Approver: Principal Engineer, Product Manager
Impact: Medium
Context
LO Dashboard needs real-time updates when borrowers complete application steps. Co-borrower flow needs live status updates. Need to choose between WebSocket and Server-Sent Events (SSE).
Decision
Hybrid Approach: WebSocket for bi-directional + SSE as fallback

Use WebSocket (Socket.io) for primary real-time communication with SSE fallback for environments where WebSocket is blocked (corporate proxies, older browsers).
Rationale
‚Ä¢ WebSocket provides bi-directional communication for collaborative features
‚Ä¢ Lower latency and overhead compared to polling
‚Ä¢ SSE provides reliable fallback with automatic reconnection
‚Ä¢ Socket.io library handles connection management and fallbacks automatically
‚Ä¢ Better user experience with instant updates
Implementation Details
‚Ä¢ Use Redis pub/sub for broadcasting across server instances
‚Ä¢ Implement connection pooling and automatic reconnection
‚Ä¢ Add heartbeat/ping to detect stale connections
‚Ä¢ Graceful degradation to polling if both WebSocket and SSE fail

Implemented

API Design - REST + GraphQL Hybrid

Date: 2025-11-28
Approver: VP Engineering, Lead Frontend Engineer
Impact: High
Decision
RESTful API for CRUD operations + GraphQL for complex queries
API Strategy
REST for:
‚Ä¢ Simple CRUD operations (create application, update borrower info)
‚Ä¢ Third-party webhooks (Plaid, TRUV callbacks)
‚Ä¢ File uploads and downloads
‚Ä¢ Standard HTTP semantics (GET, POST, PUT, DELETE)

GraphQL for:
‚Ä¢ LO Dashboard with complex data aggregations
‚Ä¢ Application detail views with nested relationships
‚Ä¢ Analytics and reporting queries
‚Ä¢ Mobile app with bandwidth constraints
Rationale
‚Ä¢ REST is simple, well-understood, and cacheable
‚Ä¢ GraphQL eliminates over-fetching and under-fetching
‚Ä¢ GraphQL provides strongly-typed schema and introspection
‚Ä¢ Frontend can request exactly what it needs
‚Ä¢ Hybrid approach uses best tool for each use case
API Standards
‚Ä¢ OpenAPI 3.0 specification for REST endpoints
‚Ä¢ GraphQL schema with Apollo Server
‚Ä¢ Versioning: /api/v1 for REST, schema evolution for GraphQL
‚Ä¢ Consistent error handling with RFC 7807 Problem Details
‚Ä¢ JWT authentication for both REST and GraphQL

Mobile Strategy - Responsive Web vs Native Apps

Date: 2025-12-05
Approver: VP Product, VP Engineering
Impact: High
Context
Research shows 67% of borrowers start mortgage applications on mobile devices. Need to decide between responsive web, progressive web app (PWA), or native iOS/Android apps.
Decision
Phase 1: Mobile-First Responsive Web Application
Phase 2: Progressive Web App (PWA)
Phase 3: Evaluate Native Apps based on adoption
Rationale
‚Ä¢ Single codebase reduces development and maintenance costs
‚Ä¢ No app store approval process - faster iterations
‚Ä¢ Progressive enhancement from web to PWA to native
‚Ä¢ PWA provides offline capabilities and push notifications
‚Ä¢ Modern web APIs support camera, biometrics, and other native features
‚Ä¢ 80% of functionality achievable with web technologies
‚Ä¢ Native apps only if specific features require them (e.g., deep OS integration)
PWA Features
‚Ä¢ Service worker for offline support
‚Ä¢ Web app manifest for installability
‚Ä¢ Push notifications for application updates
‚Ä¢ Camera access for document capture
‚Ä¢ Biometric authentication (Face ID, Touch ID via WebAuthn)
‚Ä¢ Home screen icon and splash screen
Success Metrics for Native App Decision
‚Ä¢ Mobile web traffic exceeds 75% of total traffic
‚Ä¢ App store presence provides competitive advantage
‚Ä¢ Specific native features provide 10%+ conversion improvement
‚Ä¢ Budget allocated for iOS/Android development teams

Reversed Decisions

Decision Reversals & Learnings
D-R1: Initially chose MySQL, switched to PostgreSQL
Reversed after 2 weeks of architecture planning
Rationale: PostgreSQL's JSONB and Row-Level Security features were critical for multi-tenancy. MySQL lacked these capabilities.
Learning: Thoroughly evaluate database features against architectural requirements before committing.

D-R2: Planned full native mobile apps, pivoted to PWA
Reversed during product planning phase
Rationale: PWA provides 90% of native functionality at 30% of development cost. ROI didn't justify native apps for MVP.
Learning: Question assumptions about "needing" native apps. Modern web technologies are very capable.

D-R3: Initially planned synchronous LOS integration
Reversed during technical design
Rationale: LOS integration proved unreliable with frequent timeouts. Async pattern provides better UX and resilience.
Learning: Always design for integration failures. Third-party systems will have downtime.

Lessons Learned from Decision-Making

Early Stakeholder Involvement
Involving Compliance early in Prove and credit pull decisions prevented rework. Always include affected parties in initial discussions.
Document Options Thoroughly
Writing down pros/cons forces critical thinking. Several decisions were improved by this exercise revealing overlooked factors.
Set Decision Deadlines
Open-ended discussions drag on. Setting target decision dates kept project moving. Extended only when truly needed for research.
Prototype When Uncertain
Progressive disclosure decision was validated with quick prototype. Investment of 2 days saved weeks of debate and provided concrete data.
Be Willing to Reverse
Sunk cost fallacy is real. Reversing MySQL decision after 2 weeks felt wasteful but saved months of technical debt.
Data Over Opinions
UX research data was decisive in multiple decisions (progressive disclosure, phone verification). Invest in research to inform decisions.
