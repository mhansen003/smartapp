‚úÖ Actual Outcomes & Key Decisions

Key Finding 1: Progressive Disclosure Validation
Research confirmed progressive disclosure pattern increases completion by 134%. Users appreciated seeing only relevant questions, reduced cognitive load, and clear progress indicators. Decision: Adopt progressive disclosure as core UX pattern across all flows.
Key Finding 2: LO Collaboration Timing
LOs want early notification (25% completion) but prefer borrowers to self-complete as much as possible. Intervention point: 72-hour stall triggers LO outreach. LOs value real-time dashboard visibility more than manual notifications.
Key Finding 3: Mobile Usage Pattern
67% of borrowers start application on mobile, 42% complete entirely on mobile. Mobile users 2.3x more likely to abandon if experience isn't optimized. Decision: Mobile-first design mandatory, not optional.
Decision: Three Core Personas
Established 3 primary personas: (1) Tech-Savvy First Timer (35%, high completion, values speed), (2) Busy Professional Refinancer (45%, needs flexibility, values convenience), (3) Overwhelmed Buyer (20%, needs guidance, values support). Design decisions should optimize for Persona 2 while accommodating others.
Decision: Top 5 Pain Points to Address
Priority pain points identified through dot voting: (1) Redundant data entry across forms, (2) Unclear expectations about time commitment, (3) Loss of progress on mobile, (4) Confusion about co-borrower workflow, (5) Lack of visibility into LO activity. All must be addressed in v1.

Prioritization Frameworks Used

RICE Scoring Framework
RICE is a prioritization framework that scores features based on four factors: Reach (how many users), Impact (how much it moves the needle), Confidence (how sure are we), and Effort (how much work). Formula: RICE Score = (Reach √ó Impact √ó Confidence) / Effort

Scoring Criteria

Reach (per quarter)
0-1000+ users
Impact
3=Massive, 2=High, 1=Medium, 0.5=Low, 0.25=Minimal
Confidence
100%=High, 80%=Medium, 50%=Low
Effort (person-months)
0.5-12+ months
MoSCoW Prioritization Method
MoSCoW is a prioritization technique that categorizes features into four buckets to help teams focus on what's most important for the current release.

  • Prove identity verification
  • Plaid VOA integration

Decision: MVP Feature Set (9 Core Features)
MVP includes: (1) Prove identity verification (RICE: 450), (2) Plaid VOA (RICE: 420), (3) TRUV VOIE (RICE: 380), (4) Progressive disclosure UI (RICE: 520), (5) LO dashboard (RICE: 340), (6) Co-borrower flow (RICE: 290), (7) OTP authentication (RICE: 310), (8) Mobile-responsive (RICE: 480), (9) Point of submit (RICE: 360). Total estimated effort: 18 person-months.
Decision: Phase 2 Priorities
Phase 2 (months 7-12) focuses on enhancement features: Soft credit pull, AI assistant, LO takeover, advanced configurations, and real-time notifications. These add significant value but aren't critical for initial launch.
Decision: Deferred Features
Document upload, e-signature, and hard credit integration deferred to Phase 3 or beyond. These require significant integration work and compliance review. Current workarounds via existing systems acceptable for v1.
Key Insight: Progressive Disclosure Highest Priority
Progressive disclosure UI scored highest RICE score (520) due to massive impact on completion rates (134% improvement), reach (100% of users), high confidence (validated in research), and relatively low effort (2 person-months). This becomes foundational UX pattern for entire app.
Decision: Integration Strategy Prioritization
Prioritize third-party integrations in this order: (1) Prove (identity foundation), (2) Plaid (asset verification), (3) TRUV (income/employment), (4) Soft credit (Phase 2). Rationale: Identity must come first, assets and income equally important but Plaid contract already in place.
Decision: AI Assistant Timing
AI assistant moved to Phase 2 despite high impact score. Reasoning: Core flows must be solid before adding AI layer. Better to launch with excellent core experience than mediocre AI-enhanced experience. Reassess after MVP launch data.

‚úÖ Actual Outcomes & Key Decisions

Decision: Microservices Architecture (5 Core Services)
Adopted microservices pattern with 5 core services: (1) Identity Service (Prove integration, OTP, user mgmt), (2) Verification Service (Plaid, TRUV integrations), (3) Application Service (borrower data, workflow), (4) Notification Service (email, SMS, push), (5) Integration Service (LOS, third-party APIs). Services communicate via REST APIs and event bus.
Decision: Database Schema (42 Tables)
Comprehensive schema designed with 42 tables organized into 8 domains: Users (4 tables), Applications (6 tables), Borrowers (8 tables), Properties (5 tables), Income (6 tables), Assets (4 tables), Verifications (5 tables), System (4 tables). Full normalization with proper foreign keys, indexes, and audit fields on all tables.
Decision: AWS Serverless Architecture
Serverless-first approach using AWS Lambda for compute, API Gateway for routing, RDS Aurora PostgreSQL for database, S3 for storage, CloudFront for CDN, Cognito for authentication. Benefits: auto-scaling, pay-per-use, reduced ops overhead. Estimated monthly AWS cost: $2,500-$5,000 at initial scale.
Decision: API-First Development
API-first approach with OpenAPI 3.0 specification. All API contracts defined before implementation. Frontend and backend teams can work in parallel using API mocks. Versioning strategy: URL-based (/v1/, /v2/). Backward compatibility required for 6 months.
Decision: Integration Error Handling
Comprehensive error handling for third-party integrations: Exponential backoff retry (3 attempts), circuit breaker pattern to prevent cascading failures, fallback mechanisms for non-critical integrations, detailed error logging to CloudWatch. SLA targets: 99.5% uptime, <500ms API response time at p95.
Decision: Security Architecture
Multi-layered security: TLS 1.3 for all traffic, AES-256 encryption at rest, JWT tokens for authentication (15-min expiry), RBAC with 5 user roles, PII field-level encryption, comprehensive audit logging (all data access tracked), WAF rules for DDoS protection. Annual penetration testing and compliance audits required.
Key Insight: Real-time Architecture for LO Dashboard
LO dashboard requires real-time updates. Decision: WebSocket connections using AWS API Gateway WebSocket APIs for live updates. Fallback to polling (30-second intervals) for unsupported browsers. Event-driven architecture publishes application state changes to event bus, which broadcasts to connected LO clients.

Workshop Best Practices & Lessons Learned

Facilitation Techniques Employed
  • Liberating Structures: Used 1-2-4-All pattern for idea generation, Troika Consulting for problem-solving, 15% Solutions for action planning
  • Silent Brainstorming: Individual sticky note capture before group sharing reduced groupthink and increased participation
  • Timeboxing: Strict time limits for each activity with visible countdown timers maintained energy and focus
  • Parking Lot: Captured off-topic items for later discussion, keeping workshops on track
  • Round Robin: Ensured equal voice for all participants, preventing domination by senior stakeholders
  • Dot Voting: Democratic prioritization method that quickly achieved consensus on contentious topics
  • Working Agreements: Established ground rules at start: phones away, one conversation, respect all perspectives, bias toward action

üí° Key Learnings & Recommendations

Learning 1: Invest in Pre-Work
Workshops with robust pre-work assignments (reading, surveys, data analysis) were 3x more productive. Recommendation: Always assign meaningful pre-work, send reminders, and make pre-work completion visible to create social accountability.
Learning 2: Right People in the Room
Decision-making workshops failed when key decision-makers absent. Technical workshops succeeded with right technical depth. Recommendation: Ruthlessly prioritize attendance of required participants. Reschedule if critical people can't attend.
Learning 3: Digital-First, Even In-Person
Using Miro and digital tools even in in-person workshops created better artifacts and made remote participants feel included. Recommendation: Always use digital collaboration tools. Physical sticky notes are nostalgic but inefficient.
Learning 4: Action Item Accountability Critical
95% completion rate achieved through systematic tracking, clear ownership, and public visibility of progress. Recommendation: Treat action items like sprint commitments. Name names, set dates, track religiously, escalate delays.
Learning 5: Frameworks Provide Structure
RICE scoring, MoSCoW, risk matrices provided objective frameworks that reduced subjective debate and built consensus faster. Recommendation: Always use established frameworks. Don't reinvent prioritization methods. Frameworks create shared language.
Learning 6: Break Large Topics Into Multiple Sessions
Technical architecture workshop (7 hours) was too long and exhausting. Should have been split across 2 days. Recommendation: Maximum 5 hours for virtual, 6 hours for in-person. Complex topics deserve multiple focused sessions.
